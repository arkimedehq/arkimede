# Agentic Memory (A-MEM)

Arkimede's persistent memory turns durable facts about a user into **structured,
linked notes** with hybrid retrieval — instead of injecting every stored fact into
the system prompt on every turn. The design is inspired by
[A-MEM](https://github.com/agiresearch/A-mem) (Zettelkasten-style agentic memory),
with a conservative-evolution correction: the system links and enriches notes but
never rewrites their content behind the user's back.

Memory is opt-in per user (`autoMemoryEnabled`). Every note is owned by the user
and follows a `pending → confirmed` lifecycle, so nothing enters active memory
without the user's approval.

## Note model

A note is a durable fact plus generated metadata:

| Field | Purpose |
|-------|---------|
| `content` | The fact itself (a short, self-contained sentence). |
| `tags` | 1–3 classification labels. |
| `keywords` | Salient search terms (proper names kept verbatim). |
| `context` | One sentence describing *when* the note is useful to recall. |
| `category` | Coarse bucket: `preference` \| `profile` \| `constraint` \| `knowledge`. |
| `linkedIds` | Bidirectional links to related notes (filled by evolution). |
| `pinned` | Always injected, bypassing retrieval. |
| `scope` | `personal` \| `team` \| `org` visibility. |
| `status` | `pending` (awaiting confirmation) \| `confirmed` (active). |

Metadata is generated by the summarizer model configured in `llm_configs` — the
chat model is never used inline. For automatic extraction the metadata is produced
in the same LLM call as the fact; for manual notes it is enriched fire-and-forget
(the note is valid even if enrichment fails).

## Hybrid retrieval

Rather than injecting all confirmed notes, each turn retrieves only the relevant
ones by fusing two legs:

- **Full-text (Postgres)** — a generated `tsvector` column (`content + keywords +
  context`, `simple` config for mixed languages) with a GIN index. Near-zero cost,
  exact matches on names/codes. The query is tokenized and run through `to_tsquery`
  with **OR** semantics (`plainto_tsquery` ANDs every word and never matches a full
  question); `ts_rank` rewards notes matching more terms.
- **Vector** — each confirmed note is embedded and upserted into the `user_memory`
  collection with a `{userId, memoryId, tags, category, scope, teamId}` payload.
  Handles paraphrase and semantic matches.

The two ranked lists are merged with **Reciprocal Rank Fusion**, a relevance cutoff
drops weak vector-only candidates, and a top-K cap bounds what is injected. The
retrieval query is the last user message plus the tail of the rolling summary.

Retrieval is **resilient by construction**: if the embedding service is down the
vector leg returns empty and retrieval degrades to FTS-only, and vice versa.

**Prompt-cache friendliness:** pinned notes live in the stable, cached prompt
prefix; the per-query retrieved notes live in a separate dynamic block, so the
cached prefix is not invalidated every turn.

## Evolution

When a note is confirmed, an asynchronous job (never blocking the write) lets the
memory graph reorganize:

1. Hybrid search finds the top related notes.
2. The summarizer decides which to **link** (bidirectional), which to **enrich**
   — additively only (appends to context/tags, **never** a content rewrite) — and
   whether any are near-duplicates worth **merging**.
3. Merges are **not automatic**: they become a `pending` proposal the user
   confirms; on confirmation the originals are replaced and their links inherited.
   Links and additive enrichments apply directly (they are reversible).

Guards keep it safe: at most one evolution round per insertion (no recursive
cascades), a cap on related notes, and an audit entry per evolution. Evolution
never crosses scopes — a `personal` note never modifies an `org` one.

## Scoping

Notes carry a `scope` mirroring the platform's resource model: `personal` (default),
`team` (visible to team members), or `org` (admin-promoted, visible to everyone).
Retrieval unions the scopes accessible to the user; evolution stays within a scope.

## Agent tools

When memory is enabled the agent gets three built-in tools:

- **`save_memory`** — records an explicitly requested durable fact ("remember
  that…") as a `pending` proposal (never auto-confirmed; the user approves it from
  the memory panel).
- **`search_memory`** — hybrid retrieval on demand, beyond the top-K already
  injected into the prompt.
- **`search_conversations`** — episodic recall: full-text search over past message
  transcripts (`messages.content` tsvector + GIN), scoped to the chats the user can
  access, returning excerpts with the chat title and date. No LLM in the loop. This
  is the raw episodic memory ("when did we talk about X weeks ago?") that curated
  notes do not cover.

## UI

The Settings **Memory** section shows the note list (with pending proposals to
confirm), per-note metadata, pin and scope controls, and a **graph view** where
nodes are notes (coloured by category) and edges are `linkedIds` — making the
evolution job's output visible and correctable.

## Design principles

- The summarizer model does all LLM work; the chat model is never called inline.
- Cross-provider by rule: extra context goes into the system prompt, never as extra
  messages.
- Evolution costs are observable from day one in `llm_calls` (`origin=system`,
  background priority).
- Automatic extraction captures **user facts only**; operational knowledge
  (clients, configs, projects) enters memory only explicitly via `save_memory` or
  by confirmation.

## Out of scope (for now)

- Per-agent episodic memory for multi-agent teams (same infra, `{agentId}` payload).
- Automatic forgetting/decay.
- A working/long-term hierarchy (the rolling summary already covers working memory).
